1. ما هو RTK Query
 ولماذا نحتاجه؟
في العادة، عندما تتعامل مع 
React و Laravel،
 ستحتاج للقيام بالآتي يدوياً:

كتابة useEffect لجلب البيانات.

إنشاء حالة loading (هل البيانات قادمة؟).

إنشاء حالة error (هل فشل الاتصال؟).

إنشاء حالة data لتخزين ما وصل من Laravel.

التعامل مع التخزين المؤقت (Caching) لكي لا نطلب نفس البيانات كل ثانية.

RTK Query تفعل كل هذا بدلاً منك تلقائياً!

=====================================================================

المفاهيم الأساسية (Terminology)
قبل الكود، افهم هذه المصطلحات:

Store: المخزن الرئيسي لكل بيانات التطبيق.

API Slice: هو المكان الذي نُعرّف فيه الروابط (Endpoints) الخاصة بـ Laravel.

Queries: تُستخدم لـ جلب البيانات (GET).

Mutations: تُستخدم لـ تعديل البيانات (POST, PUT, DELETE).

Tags: نظام ذكي لإخبار RTK Query متى يجب عليه "تحديث" البيانات تلقائياً.

=====================================================================

In Store.js

لماذا نضع هذه المكونات بالتحديد؟

[apiSlice.reducerPath]: هذا الجزء يخبر 
Redux 
أن كل البيانات التي تأتي من الـ 
API 
(الكاش، حالة التحميل، البيانات) 
يجب أن تُخزن في مكان محدد داخل الـ 
Store.

authReducer: نحتاجه لأننا نريد تخزين بيانات المستخدم 
(مثل الاسم والرتبة)
 بشكل دائم في الـ 
State 
لنصل إليها من الـ Sidebar أو الـ ProtectedRoute
 دون الحاجة لطلبها من السيرفر في كل مرة.

apiSlice.middleware: هذا هو "المحرك" الذي يجعل 
RTK Query يعمل.
 هو المسؤول عن مراقبة الطلبات،
 وإعادة جلب البيانات تلقائياً عند تغيير الـ 
 Tags،
 ومعالجة الـ 
Re-auth
(التجديد التلقائي للتوكن).
